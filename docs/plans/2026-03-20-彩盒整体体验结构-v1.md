## 1. 这份 v1 解决什么问题

`v0` 解决的是：

- 产品愿景
- 情感内核
- 首页与主路径前半段骨架
- `See` 的三层状态

但在进入原型阶段后，我们优先验证了前几页的成立性，暂时没有把后半程和整体产品结构系统补完。

这份 `v1` 的目标就是把那部分补齐，重点回答 4 个问题：

1. `Try -> 收进彩盒` 之后到底发生什么  
2. `我的彩盒` 作为一个完整世界，最小结构应该是什么  
3. `自然样本精选集` 在整体产品中的长期位置是什么  
4. 什么时机适合把最小 MVP 工程化，真正跑到手机上验证手感

这份文档不推翻 `v0`，而是在其基础上补完整体闭环。

## 2. 当前已经被验证过的前半段

截至目前，已经有足够把握的前半段主路径是：

`Home -> See / Live -> See / Frozen -> See / Picked -> Try`

这一段已经证明了几件关键事情：

- 首页不是功能大厅，而是状态承接页
- 主入口需要先进入实时观察，而不是直接进入取色
- `停住这一刻` 是主入口进入取色前必须存在的桥
- `See` 必须是同一个观察空间里的连续状态，而不是几张割裂页面
- `Try` 不应是投放器或教具，而是颜色关系工作台

因此，`v1` 不再重新发明前半程，而是在此前提下补完后半程。

## 3. 完整主路径 v1

`彩盒` 的最小完整体验闭环，收口为：

`Home -> Live/Frozen/Picked -> Try -> 收进彩盒 -> 在我的彩盒里回看这一页`

这条路径还隐含着一个节奏判断：

- 它本质上不是为“连续快速采集”优化的
- 而是为“一次观察，形成一页发现”优化的

也就是说，`彩盒` 的主路径更鼓励：

- 停下来
- 看一会儿
- 带回几种颜色
- 让它们待一会儿
- 再把这一页收进去

而不是：

- 连续拍
- 连续提
- 快速扫过很多对象

如果未来要支持更高频、更流动的采集，它也应作为单独的旁路能力存在，而不应破坏这条主路径的节奏。

这一条路径的每一段职责如下：

### 3.1 Home

承担：

- 开始新的观察
- 回到未完成的现场
- 从精选样本开始
- 从相册带进来
- 看见自己已经拥有的一点记录

### 3.2 See

承担：

- 观察
- 定格
- 取色
- 替换、移除、切换当前编辑色格
- 决定何时把颜色带去 `Try`

### 3.3 Try

承担：

- 让已经带回来的颜色彼此待一会儿
- 看见关系，而不是急着得出结论
- 允许用户收下当前这一页

### 3.4 收进彩盒

承担：

- 把本次观察与尝试收成一页完整记录
- 给予温和完成感
- 让这一页进入 `我的彩盒`

### 3.5 我的彩盒

承担：

- 回看
- 重遇
- 比较
- 迟到地意识到自己的认知正在成长

## 4. `收进彩盒` 之后发生什么

这一段是 `v1` 的关键补完。

`收进彩盒` 不是“保存文件”，也不是“完成任务”。它更像：

- 把今天这一点发现轻轻放进自己的小盒子

因此，点击 `收进彩盒` 后，不应该出现：

- 工具式保存成功提示
- 表单式命名弹窗
- 奖励式勋章反馈
- 完成打勾页

更合适的结果是一个 `轻收纳状态`：

### 4.1 收纳反馈

反馈应当服务于“这一页被接住了”，而不是“系统保存成功了”。

建议方向：

- 当前页轻轻收拢
- 颜色关系结果和来源对象一起被收成一张页片
- 文案极轻，比如：
  - `这一页收进彩盒了`
  - 或不出现文案，只用视觉完成收纳感

### 4.2 收纳后的两个可达方向

收进之后，产品不应立刻把用户踢回首页，而应给两个自然方向：

1. `回到我的彩盒看看这一页`
2. `继续新的观察`

但这两个方向都不应显得像强选择题。更合理的是：

- 默认轻轻落到 `我的彩盒` 中新收进的一页
- 同时顶部或附近保留回到观察的自然路径

这样，收进彩盒不会像流程终点，而更像：

- 从观察空间，走进自己的记录空间

### 4.3 收纳后的情绪目标

进入 `我的彩盒` 后，情绪上不能只留下：

- `终于收进来了，可以歇会儿了`

因为这会把主路径做成一个“完成任务、退出流程”的结束点。

更合适的情绪目标是：

- 先让用户感到“这一页已经属于我了”
- 再轻轻保留一点：`不妨再来一次`

也就是说，进入拥有态之后，仍应留有轻量热度，而不是只剩如释重负。

### 4.4 `彩盒 / 新收页打开态`

`收进彩盒` 之后的默认落点，应定义为：

- `彩盒 / 新收页打开态`

它不是列表页，也不是一般的单页回看，而是：

- 刚刚被收进来的这一页
- 第一次在 `彩盒` 世界里被打开

这个状态至少要同时承担两件事：

1. `拥有感`
   - 用户明确知道：
     - 刚才那一页没有消失
     - 它已经被整理成“我的这一页”

2. `轻继续感`
   - 页面里保留一个不喧宾夺主的继续观察入口
   - 让用户感到：
     - 不是流程结束了
     - 而是“这一页先收好了，如果愿意，还可以再去看一页”

因此，这一页的结构应倾向于：

- 上部：当前页的来源对象与颜色关系结果
- 中部：这页的颜色、来源、时间感
- 下部：一个很轻的继续观察入口

而不应倾向于：

- 成功提示页
- 奖励页
- 列表中转页

### 4.5 草图验证后的新增判断

针对 `收进彩盒 -> 收拢中 -> 新收页打开态` 的低保真草图验证，当前已经形成以下结论：

- `Try` 本身就是“收进前”的起点态，不需要为“收之前”单独再设计一张重复页面
- `收拢中的中间态` 是必要的，但它的职责不是说明流程，而是承担：
  - 从 `Try` 的工作台态切出
  - 把当前页收成一张页片
  - 把用户心智从“过程态”带向“拥有态”
- 这个中间态目前可以用草图确认结构，但真正成立仍然依赖后续动效
- `新收页打开态` 已经能够被确认为默认落点：
  - 用户不应先回到首页
  - 也不应先被扔进列表
  - 而应先看见“刚刚收下的这一页”

草图也同时证明了两件事：

1. 有些信息可以通过草图确认：
   - 页面职责分工
   - 第一眼看什么
   - 视线顺序
   - “拥有感在前，继续感在后”的层级关系

2. 有些体验不能靠草图完全确认：
   - `收拢中的中间态` 是否真正成立
   - “这一页被收住了”的手感
   - 从过程态切入拥有态的节奏感

这些部分后续必须通过动效和真实运行手感来验证，而不能指望靠静态页完全确定。

### 4.6 `新收页打开态` 与 `单页回看` 的关系

这两者不能被做成两套系统，但也不能被误认为同一状态。

它们的共同骨架应保持一致：

- 上部：来源对象与这一页的核心内容
- 中部：色组与关系结果
- 下部：轻身份信息与轻继续入口

也就是说，用户无论是“刚刚收进来”还是“以后回来看这一页”，都应当认得出：这是同一页。

但它们的时态与职责不同：

#### `新收页打开态`

它是一个过渡态，职责是：

- 承接 `Try -> 收进彩盒`
- 明确告诉用户：这一页现在已经属于彩盒了
- 把心智从过程态切入拥有态

因此，它会比常规单页回看多一点“刚刚发生”的新鲜感：

- 更轻地提示“现在它已经在彩盒里了”
- 更明显地保留一点刚收进来的在场感

#### `单页回看`

它是一个稳定态，职责是：

- 从页列表点开任意一页时，稳定地回看这一页
- 不再承担“刚刚被收进来”的过渡任务
- 重点放在“这页本身是什么”

因此，它不需要再强调：

- “刚刚收下”
- “现在已经在彩盒里”

而应更自然地成为：

- 我的一页发现记录

一句话收口：

- `新收页打开态`：从过程切入拥有
- `单页回看`：在拥有态里稳定地回看一页

## 5. `我的彩盒` 的最小结构

`我的彩盒` 在 `v1` 里不再只是“一角”概念，而要至少定义清楚它的最小信息结构。

### 5.1 它不是素材库

它不是：

- 相册
- 文件夹
- 只有色卡没有过程的结果页

### 5.2 它最小的单位是一页完整记录

每一页至少包含：

- 来源对象
  - 现场拍的
  - 从相册带进来的
  - 从精选样本开始的
- 取到的颜色
- 是否进入过 `Try`
- 当前被收下来的颜色关系结果
- 时间感或主题线索

这意味着：

- 用户自己的素材和“玩出来的颜色”
- 不会分散在两个系统里
- 而是共同存在于同一页记录中

### 5.3 `我的彩盒` 的最小信息层

在 MVP 阶段，`我的彩盒` 只需要 3 层：

1. `页列表`
   - 以页片形式浏览自己收进来的记录

2. `单页回看`
   - 能看到这页的来源对象、取到的颜色、关系结果

3. `再次进入`
   - 从这页重新进入观察或 `Try`，暂不强求

MVP 阶段不需要先做：

- 复杂筛选
- 自动分类体系
- 主题馆
- 成长统计面板

## 6. 自然样本精选集的长期位置

这是 `v1` 里另一块需要明确补完的结构。

`自然样本` 不是一次性 onboarding，也不是用户自己的素材集合。它更适合被定义为：

- 一个可以持续扩充的精选集

它长期承担 3 个角色：

### 6.1 启动角色

- 第一次使用时的低门槛开始
- 不方便现场观察时的兜底入口

### 6.2 教学角色

- 用更适合观察的对象，帮助孩子理解：
  - 一个对象里不止一种颜色
  - 颜色可以被带回来
  - 颜色之间可以待一会儿

### 6.3 运营角色

- 将来产品更新、节气专题、季节精选、编辑推荐的载体

但它始终不是“我的东西”本身。只有当用户真的和某个样本走完一次过程后，那一页才可以进入 `我的彩盒`。

这和那条原则完全一致：

`不能伪造拥有感`

## 7. 全局信息结构 v1

在 `v1` 里，信息结构可以更明确地收成：

### 7.1 首页

承担状态承接与开始。

### 7.2 See

承担观察、定格、取色、编辑。

### 7.3 Try

承担颜色关系工作台。

### 7.4 我的彩盒

承担个人记录空间。

### 7.5 精选样本

在信息架构上，它不是和 `我的彩盒` 平级的长期主世界，而更像：

- 首页上的一个重要开始入口
- 或以后 `发现` / `探索` 维度中的一部分

这意味着，在 MVP 阶段，不需要为了精选样本单独造一整个复杂频道页；只需确保：

- 首页可进
- 样本能进入主路径
- 样本走完后能进入 `我的彩盒`

## 8. AI 伙伴在整体结构里的位置

`v0` 已定义了 AI 的双层结构，`v1` 补充的是它在整条主路径中的出场节奏。

### 8.1 首页

- 不需要一打开就“跳出来说话”
- 更适合存在于气息层和极轻提示中

### 8.2 See / Live

- 不主动说话
- 让观察先发生

### 8.3 See / Frozen / Picked

- 这是 AI 最适合轻触发的地方
- 例如：
  - `原来这里不止一种绿色`
  - `最亮的地方和最暗的地方一样吗`

### 8.4 Try

- AI 仍应克制
- 它可以轻轻命名感受，但不急着解释原理

### 8.5 我的彩盒

- 回看时，AI 可以成为一个更轻的整理者或提醒者
- 例如提示：
  - 你最近收了很多天空的颜色
  - 你这阵子好像很喜欢叶子的绿色

但这应属于后续阶段，不进入 MVP 核心。

## 9. MVP 的边界

现在已经可以比较清楚地定义第一阶段 MVP。

### 9.1 MVP 必须包含

1. 首页
2. 主入口进入 `See / Live`
3. `Live -> Frozen -> Picked`
4. `Try`
5. `收进彩盒`
6. `我的彩盒` 中最小的页列表和单页回看

### 9.2 MVP 可以先不包含

- 复杂 AI 伙伴对话
- 精细的颜色文化命名层
- 丰富的样本馆
- 多维度检索和分类
- 复杂动效系统
- 跨页重编辑能力

### 9.3 MVP 工程化的最佳时机

不在现在，也不必等整个产品设计完。

最合适的时机是：

- `收进彩盒` 这一动作的内容、反馈、落点和 UI 被定义清楚之后
- 同时 `我的彩盒` 的最小回看页被做出来之后

因为只有到这时，最小闭环才真正闭上：

- 看到了
- 带回来了
- 放在一起看了
- 收起来了
- 还能回看

这时把 MVP 工程化到手机上，才是在验证真实手感，而不是只把半条流程搬到设备上。

## 10. 下一步建议

现在不建议继续无限细修现有前几页。

更合理的顺序是：

### 10.1 先把这份 `v1` 作为整体结构锚点

用于约束后续：

- 首页专题
- `See / Try` 专题
- `收进彩盒` 之后的页面设计
- MVP 工程实现边界

### 10.2 接着优先补两块

1. `收进彩盒` 的内容、反馈、页面落点
2. `我的彩盒` 的最小列表页和单页回看页

### 10.3 再决定是否进入 MVP 工程化

如果这两块成立，就可以开始判断：

- 是否要把最小 MVP 跑到手机上
- 用真实设备去验证：
  - 观察节奏
  - 按钮位置
  - 手势与点击的手感
  - 页面切换是否仍然轻

## 11. v1 的一句话总结

`v0` 证明了前半段主路径能成立，`v1` 则把“收进彩盒之后的世界”正式补上了。
